# SaaS管理ツール POC 運用ルール（案）

---

## 1. POC運用の位置づけ

| 項目 | 内容 |
| --- | --- |
| 期間 | POC構築後2〜4週間のトライアル |
| 対象 | 先行導入部署（1〜2部署）の管理者・メンバー |
| 目的 | 自社開発ツールで現場課題を解決できることを実証する |
| ゴール | デモ＋トライアルの結果を踏まえ、本番移行（全社展開）を判断 |

## 2. 運用体制（POC）

| 役割 | 担当 | 主な責務 |
| --- | --- | --- |
| ツール管理者（admin） | 自分＋バルくん | ツール全体の管理、デモ実施、フィードバック収集 |
| 部門管理者（manager） | 先行導入部署の管理者 | 自部門のSaaS登録・サーベイ回答・タスク管理 |
| 一般ユーザー（viewer） | 先行導入部署のメンバー | 自分のSaaS利用状況の閲覧、サーベイ回答、申請 |

## 3. SaaS台帳の運用

### 3.1 初期データ投入（POCセットアップ時）

| データ | 投入方法 | 件数目安 |
| --- | --- | --- |
| SaaS一覧 | CSV一括取込（既存Excel等から変換） | 先行部署分30〜50件 |
| アカウント情報 | Entra ID同期（自動）+ CSV取込（手動） | 先行部署分 |
| メンバー情報 | Entra IDユーザー同期バッチを手動実行 | 全社員分（退職者検出のため） |

### 3.2 SaaS導入申請フロー（POCで実証）

```
導入希望者
  ↓ ツールの申請フォームからSaaS利用申請
    （SaaS名、用途、利用人数、費用概算）
情シス（承認者）
  ↓ Teams通知で承認依頼を受信
  ↓ ツールの承認画面で承認 or 却下
承認された場合
  ↓ 申請者に承認通知
  ↓ SaaS台帳に自動登録（or 管理者が登録）
  ↓ アカウント発行（各SaaS管理画面で手作業）
  ↓ ツールにアカウント情報を登録
利用開始
```

## 4. アカウント管理の運用

### 4.1 アカウント登録方法

| 方法 | 用途 | POCでの対応 |
| --- | --- | --- |
| Entra ID同期（自動） | M365ユーザー情報の自動取得 | ○ 手動実行ボタン |
| CSV取込（手動） | API非対応SaaS（260件の大半） | ○ |
| 手動登録 | 少数のアカウント登録 | ○ |
| SaaS API連携（自動） | Google Workspace / Slack等 | × POC範囲外 |

### 4.2 退職時のフロー（POCで実証）

```
人事から退職情報を受領
  ↓ Entra IDでアカウント無効化（accountEnabled=false）※既存フロー
  ↓ ツールの管理画面から「Entra IDユーザー同期」を手動実行
  ↓ 退職者アカウント検出が自動起動 → 残存アカウント一覧を表示
  ↓ タスク管理画面で「退職処理」タスクを自動生成
    □ Google Workspace アカウント削除（担当: ○○）
    □ Slack アカウント削除（担当: ○○）
    □ SUUMO管理画面 アカウント削除（担当: △△）
    □ ...
  ↓ 各担当者がSaaS管理画面でアカウント削除（手作業）
  ↓ ツール上でタスクを「完了」にチェック
```

### 4.3 入社時のフロー

```
人事から入社情報を受領
  ↓ Entra IDにアカウント作成（既存フロー）
  ↓ ツールの管理画面から「入社処理」タスクを生成
    □ Google Workspace アカウント作成（担当: ○○）
    □ Slack アカウント招待（担当: ○○）
    □ 業務系SaaS アカウント作成（担当: △△）
    □ ...
  ↓ 各担当者がSaaS管理画面でアカウント作成（手作業）
  ↓ ツール上でアカウント情報を登録 + タスクを「完了」にチェック
```

## 5. 棚卸しの運用（POCで実証）

### 5.1 アカウントサーベイのフロー

```
管理者がサーベイを作成（対象SaaS・対象メンバーを選択）
  ↓ 対象メンバーにTeams通知（回答URLリンク付き）
メンバーが回答画面でSaaS利用状況を回答
  - 「利用中」→ そのまま
  - 「利用していない」→ 削除候補にマーク
  ↓ 3日後・7日後に未回答者にリマインド通知
管理者が結果画面で確認
  ↓ 「利用していない」アカウント → 削除タスクを生成
```

### 5.2 PW更新リマインダーのフロー

```
管理者がPW更新サーベイを作成（対象SaaS・90日サイクル等を設定）
  ↓ 対象メンバーにTeams通知（「○○のPWを更新してください」）
メンバーが各SaaSでPWを更新後、ツール上で「更新済み」を回答
  ↓ 未対応者にリマインド
管理者が対応状況を確認
```

## 6. 通知

### 6.1 通知チャネル（POC）

| 通知先 | 用途 | 実装 |
| --- | --- | --- |
| Teams | アラート・承認依頼・サーベイ配信・タスク通知 | Power Automate Workflows Webhook |

POCではTeams通知のみ。メール通知は本番移行時に追加。

### 6.2 通知一覧

| 通知 | タイミング | 通知先 |
| --- | --- | --- |
| 退職者アカウント検出 | Entra ID同期実行後 | 情シス |
| サーベイ配信通知 | サーベイ作成時 | 対象メンバー |
| サーベイリマインド | 配信3日後・7日後 | 未回答者 |
| タスク生成通知 | タスク作成時 | 担当者 |
| タスク期限超過通知 | 手動実行 | 担当者 |
| 承認依頼通知 | 申請時 | 承認者 |
| 承認結果通知 | 承認/却下時 | 申請者 |

## 7. POCトライアルの進め方

### 7.1 ステップ

| ステップ | 期間 | 内容 |
| --- | --- | --- |
| 1. デモ | 1日 | バルくん・先行導入部署の管理者にデモ実施 |
| 2. 初期設定 | 2〜3日 | 先行部署のSaaS棚卸し→台帳登録→アカウント登録 |
| 3. トライアル | 2〜4週間 | 先行部署で実運用。サーベイ・タスク管理・申請承認を実際に使う |
| 4. フィードバック | 1週間 | ヒアリング、改善点の洗い出し |
| 5. 判断 | - | POC結果を踏まえ、本番移行（全社展開）の可否を決定 |

### 7.2 フィードバック収集項目

- サーベイ（棚卸しアンケート）は使いやすいか
- タスク管理（入退社チェックリスト）は役に立つか
- 申請・承認フローはメール+Excelより楽か
- 退職者アカウント検出は正しく動くか
- 台帳の登録・更新は手間が許容範囲か
- DXECOと比べて不足している機能はあるか
- 追加で欲しい機能はあるか

### 7.3 判断基準

| 判断 | 条件 |
| --- | --- |
| **本番移行GO** | 現場管理者から「これで業務が回る」と評価される。DXECOにない差別化機能（承認フロー・退職者自動検出）が高評価 |
| **追加開発して再トライ** | 基本は良いが特定機能の不足がある。追加開発後に再トライアル |
| **DXECO検討に戻る** | 開発リソースの継続確保が困難、または想定外の要件が大量に出た場合 |

## 8. 本番移行時に追加する運用

POCでは省略し、本番移行時に整備する項目。

| 項目 | 内容 |
| --- | --- |
| 管理者の引き継ぎルール | 主担当＋副担当の2名体制 |
| 定期棚卸しスケジュール | 四半期ごと（1月・4月・7月・10月） |
| バッチ自動スケジュール | Solid Queueのrecurring tasksで定期実行 |
| 契約更新アラート（日次自動） | 更新1ヶ月前・1週間前に自動通知 |
| メール通知 | Teams + メールの併用（見逃し防止） |
| シークレットローテーション手順 | Entra IDシークレットの定期更新手順 |
| データエクスポート | 棚卸しレポート・コストレポートのCSV/PDF出力 |
| コストダッシュボード | 月次コスト推移グラフ・部門別コスト |
