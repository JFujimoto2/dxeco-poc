# SaaS管理ツール「dxeco（デクセコ）」調査レポート

～ DXECO vs 自社開発POC 最終判断 ～

## 1. 製品概要

dxeco（デクセコ）は、株式会社オロ（東証プライム上場）が提供するSaaS管理ツール。

| 項目 | 内容 |
| --- | --- |
| 提供元 | 株式会社オロ |
| 公式サイト | https://www.dxeco.io/ |
| SaaSデータベース | 3,500件以上のSaaS対応 |

### 料金体系（見積もり確認済み）

| 項目 | 金額 |
| --- | --- |
| 基本料金 | 年36万円 / 100名単位 |
| 監査ログ（オプション） | 年12万円 |
| IP制限（オプション） | 年12万円 |
| 契約単位 | 100名刻み、途中減額不可 |

| 規模 | 年額 | 月額換算 | 1アカウント単価 |
| --- | --- | --- | --- |
| 500名（オプションなし） | 180万円 | 15万円 | 300円 |
| 500名（オプションあり） | 204万円 | 17万円 | 340円 |
| 400名（オプションあり） | 168万円 | 14万円 | 350円 |
| 5年累計（500名・オプションあり） | **1,020万円** | - | - |

## 2. 現場の実態（当初想定とのギャップ）

| 項目 | 当初想定 | 現場実態 |
| --- | --- | --- |
| 管理対象SaaS数 | 数十件 | **約260件** |
| 利用ユーザー数 | 数十名 | **約500名** |
| API連携可否 | 主要SaaSは可能 | **大半が連携不可** |
| 管理ニーズ | 台帳＋コスト可視化 | **アンケート・タスク管理・申請承認が最重要** |

管理対象の大半は不動産業界特化サービス（SUUMO、楽待、健美家等のポータルサイト、保証会社サイト、ローン審査サイト）やSNS分析ツール（GA、Clarity、LINE、Facebook等）で、API連携は原則不可能。

## 3. DXECO機能の評価（製品資料 2024年12月版より）

### 3.1 機能構成（Discover / Insight / Action）

| カテゴリ | 機能 | 内容 |
| --- | --- | --- |
| Discover | 契約SaaS管理 | SaaS台帳、契約情報、カスタムフィールド |
| Discover | アカウント一覧 | 従業員×SaaSマトリクス、CSV取込 |
| Discover | アカウントサーベイ | アンケート形式の棚卸し（利用可否を配信・回収） |
| Discover | シャドーIT管理 | ブラウザ拡張（3,500ツール対応） |
| Discover | IT資産管理 | PC・携帯等の備品台帳 |
| Discover | カスタム連携 | **有償開発（オロ社による追加費用）** |
| Insight | 退職者検知 | 退職者の検知（※台帳と非連動） |
| Insight | 新規アカウント検知 | 新規登録アカウントの検知 |
| Insight | 不要アカウント管理 | 未利用アカウントの管理 |
| Insight | 契約更新アラート | 更新期限の通知 |
| Action | タスク管理 | **チェックリスト機能（※承認フローではない）** |
| Action | Zapier連携 | 外部ツール連携 |
| Settings | カスタムロール | 権限管理 |

### 3.2 タスク管理機能の実態（★重要発見）

**現場の期待:** 「SaaS利用申請→上長承認→自動実行」のワークフロー

**製品資料で確認できた実態:**
- 入社/退職/異動タスクのプリセットからチェックリストを自動生成
- 各タスクに担当者・期限・メモ・ステータス（完了/未完了）を設定
- タスク例: アカウント削除、PC返却、社用携帯返却、入館証返却

**確認できなかったこと:**
- 「申請→上長承認→自動実行」のワークフロー機能
- 承認後のSaaSアカウント自動作成・削除

つまりDXECOの「タスク管理」は承認フローではなく**チェックリスト**。現場が最も求めている申請・承認フローは、製品資料上で確認できなかった。

## 4. 現場期待とDXECO機能のギャップ

| 現場の期待 | DXECOの実態 | ギャップ |
| --- | --- | --- |
| 申請・承認フローがツール上で完結 | タスク管理（チェックリスト）のみ | **大** |
| アカウント付与・削除がツールで完結 | API連携SaaSのみ読取可、書込不可 | **大** |
| 退職者アカウントを自動で検出 | 検知と台帳が非連動、手作業突合せ | **大** |
| Entra ID連携で組織情報を自動反映 | CSV手動インポート | **大** |
| PW更新アラートで更新確認 | アンケート自己申告収集 | 中 |
| 棚卸しアンケート効率化 | アカウントサーベイ対応可 | なし |
| タスク管理で作業フロー定型化 | プリセットチェックリスト生成可 | 小 |
| 管理者異動時引継ぎ容易 | DXECO内データで引継可 | なし |
| API連携不可でも登録・管理可 | 任意登録+CSV取込対応 | なし |

## 5. DXECO実機評価で判明した技術的制約

### 5.1 ブラウザ拡張（シャドーIT検知）の制約

| 課題 | 詳細 |
| --- | --- |
| MSプロファイル識別非対応 | ChromeでMSプロファイルベースのユーザー識別ができない |
| SaaSログインアカウント不明 | どのSaaSに「どのアカウントで」ログインしているか追えない |
| 検知結果と台帳が非連動 | 利用状況検知とアカウント台帳が独立、手作業突合せが必要 |
| 精度の課題 | URLクリック＝利用とみなされ、誤検知が発生 |

### 5.2 台帳管理・API連携の制約

| 課題 | 詳細 |
| --- | --- |
| 退職者アカウント自動検出なし | Entra ID連携がないため退職者の自動検出不可 |
| API連携対象SaaSが限定的 | 260件の大半（不動産業界特化サービス）は対象外 |
| 台帳→SaaSへの書き戻し不可 | アカウント作成・削除は各SaaS管理画面で手作業 |
| カスタム連携は有償開発 | オロ社による追加費用が発生 |

## 6. コスト比較

### 6.1 5年間のコスト推計

| 期間 | DXECO（500名・オプション込） | 自社開発POC→本番 |
| --- | --- | --- |
| 初期（POC） | - | 開発工数2週間 |
| 1年目 | 204万円 | インフラ6万円 + 保守工数 |
| 3年累計 | 612万円 | インフラ18万円 + 保守工数 |
| 5年累計 | **1,020万円** | インフラ30万円 + 保守工数 |

### 6.2 DXECOのコスト構造の問題

- セキュリティ機能（監査ログ・IP制限）が有料オプション → 個人情報を扱うツールでは事実上必須（年24万円追加）
- 100名刻み・途中減額不可 → 段階的導入との相性が悪い
- カスタム連携は有償開発 → 不動産業界特化サービスとの連携が必要になった場合に追加費用
- 価格は資料非掲載（要問い合わせ）→ 透明性が低い

### 6.3 自社開発のコスト

| 項目 | 費用 |
| --- | --- |
| インフラ（Azure App Service + PostgreSQL） | 月額約5,000円 |
| 開発工数（POC） | 約2週間（1名） |
| 保守工数（本番） | 月数時間程度 |
| セキュリティ機能 | **標準装備（追加費用ゼロ）** |

## 7. 結論: 自社開発POCを先行実施

### 7.1 判断理由

| # | 理由 |
| --- | --- |
| 1 | DXECOの年204万円に対し、**現場が最も求める「申請・承認フロー」が製品資料で確認できない** |
| 2 | 管理対象260件の大半がAPI連携不可であり、**DXECOのAPI連携・シャドーIT検知の価値がほぼ活かせない** |
| 3 | 自社開発なら**Entra ID連携で退職者自動検出**（DXECOにない機能）が実現できる |
| 4 | セキュリティ機能（IP制限・監査ログ）が自社開発なら標準装備、**DXECOは年24万円の有料オプション** |
| 5 | 5年間の累計コストが**約1,000万円 vs インフラ30万円＋保守工数**で圧倒的な差 |
| 6 | POCで実証してから判断でき、**リスクが小さい** |

### 7.2 POCで証明すること

| # | 機能 | DXECOとの比較 |
| --- | --- | --- |
| 1 | アンケート（棚卸し・PW更新確認） | DXECOと同等 |
| 2 | 入退社タスク管理（チェックリスト） | DXECOと同等 |
| 3 | **SaaS利用申請・承認フロー** | **DXECOにない（★差別化）** |
| 4 | **退職者アカウント自動検出（Entra ID連携）** | **DXECOにない（★差別化）** |
| 5 | IP制限・監査ログが標準装備 | DXECOは有料オプション |

### 7.3 バルくんへの確認事項（優先度順）

| # | 確認事項 | 背景 |
| --- | --- | --- |
| 1 | DXECOの「タスク管理」で申請承認は可能か | 製品資料ではチェックリストのみ確認。オロ社に要確認 |
| 2 | POC先行開発のGO判断 | 2週間で作れる前提で、先にPOCを見せて判断する進め方 |
| 3 | 先行導入部署の選定 | POCトライアルの対象部署 |
| 4 | セキュリティオプションの見積もり確認 | 監査ログ・IP制限が見積もりに含まれているか |

### 7.4 リスクと対策

| リスク | 対策 |
| --- | --- |
| POC開発が想定以上に長引く | AI活用で高速開発。2週間で最小限のPOCに絞る |
| 現場からDXECOにしかない機能が求められる | POCフィードバックで早期に判明。必要なら追加開発 |
| 保守・運用リソースの継続確保 | 開発者が異動した場合のリスク → ドキュメント整備で対応 |
| POC結果がDXECO以下の評価 | その場合はDXECO導入に戻る（判断材料が増えた状態で） |
