## 4. DXECOの認証方式の問題点

DXECOのログインはID/パスワード方式であり、SaaS管理ツールとしてセキュリティ上の懸念がある。

| 観点 | DXECO（ID/パスワード） | 自社開発（Entra ID SSO） |
|------|----------------------|------------------------|
| パスワード管理 | ユーザーがDXECO専用のパスワードを管理する必要がある | 不要。Entra IDアカウントで自動ログイン |
| MFA（多要素認証） | DXECO側で対応しているか不明 | Entra IDの条件付きアクセスポリシーでMFA強制可能 |
| 退職者のアクセス遮断 | DXECO側のアカウントを別途削除する必要がある。削除漏れのリスク | Entra IDを無効化した瞬間にアクセス不可 |
| アクセス制御 | DXECO側の機能に依存 | Entra IDの条件付きアクセスポリシー（社外ブロック、デバイス制限等）がそのまま適用 |
| パスワード漏洩リスク | フィッシングやリスト攻撃の対象になり得る | パスワード自体が存在しないため漏洩リスクなし |
| 管理負荷 | DXECOのアカウント管理が追加業務になる | 既存のEntra ID運用に統合。追加の管理作業なし |

> **SaaSのアカウント情報・契約情報・個人情報を一元管理するツールのログインがID/パスワードであることは、セキュリティ上の矛盾。** 管理対象のSaaSにSSO導入を推進する立場のツールが、自身はSSOに非対応。

## 5. アカウント同期方式の根本的な違い

| 観点 | DXECO | 自社開発 |
|------|-------|---------|
| 同期の仕組み | 各SaaSのAPIに個別接続してアカウント情報を読み取り | Entra IDのエンタープライズアプリ経由で一括取得 |
| API未提供のSaaS | **同期不可。CSV手動取込のみ** | **SSO連携していればAPI不要で自動同期可能** |
| 接続設定 | SaaSごとにAPI設定が必要（接続情報の管理が煩雑） | Entra IDへの接続1本のみ。SaaS側の設定不要 |
| 対応範囲 | DXECOが対応しているSaaSのみ | Entra IDでSSO連携している全サービス |
| 業界特化SaaS | API未提供が多く、CSV手動運用になりがち | SSO連携さえしていれば自動同期可能 |

> **DXECOはSaaS側のAPI有無に依存する。自社開発はEntra ID側のデータを見ているため、SaaS側にAPIがあるかどうかは関係ない。** 不動産業界のようにAPI未提供の専門SaaSが多い環境では、この差は決定的。

## 6. DXECOのコスト構造の問題点

- セキュリティ機能（監査ログ・IP制限）が有料オプション → 個人情報を扱う以上は事実上必須
- 100名刻み・途中減額不可 → 段階的導入との相性が悪い
- カスタム連携は有償開発 → 不動産業界特化サービスとの連携は追加費用
- 価格は資料非掲載（要問い合わせ）→ 透明性が低い

---

## 7. 自社開発の優位点

- **5年間で約990万円のコスト削減**
- Entra ID SSO認証（DXECOはID/パスワード。SaaS管理ツールとしてセキュリティ上の矛盾）
- 「申請・承認フロー」の基本形を実装済み（DXECOにはない機能。実運用に合わせて承認ルート等の改修が必要）
- Entra ID連携で退職者を自動検出（DXECOはCSV手動突合せ）
- 監査ログ・IP制限が追加費用ゼロ
- 要望に応じて自由にカスタマイズ可能
- テスト・CI整備済みで保守コストが低い

## 8. 想定される懸念への回答

### 「開発者が異動・退職したら保守できないのでは？」

自社開発において最も指摘されやすい属人化リスクについて、以下の対策を講じている。

| 対策 | 内容 |
|------|------|
| ドキュメント整備 | ユーザー運用ガイド、開発運用手順書、環境変数ガイド、機能別ドキュメント（11本）を整備済み |
| Rails標準構成 | 特殊なフレームワークや独自アーキテクチャを使わず、Rails標準の構成に準拠。Railsエンジニアであれば即座にキャッチアップ可能 |
| テスト・CI | RSpec 194テスト + Playwright E2E 54テスト + CI自動実行。引き継ぎ後も安全に修正できる基盤がある |
| Docker対応 | Docker Composeで環境構築が完結。新しい担当者が即日で開発環境を立ち上げられる |
| インフラのコード化 | Azure リソース作成スクリプト（setup.sh）とGitHub Actionsデプロイワークフローを整備。手順書なしでもデプロイ可能 |

仮に外部委託で保守する場合でも、Rails + PostgreSQL + Azure という一般的な技術スタックのため、対応可能なベンダーは多い。

### 「POCから本番化するときに追加コストがかかるのでは？」

POCの段階でインフラ構成（Azure Container Apps）、CI/CD、テスト基盤、認証（Entra ID SSO）、Docker対応をすべて本番レベルで構築済み。本番化に必要な追加作業は以下の通りで、大規模な追加開発は不要。

| 作業 | 工数 |
|------|------|
| 承認フローの運用合わせ改修（多段階承認等） | 2〜3日 |
| 本番環境のAzureリソース作成 | 0.5日（スクリプト実行） |
| 本番データの初期投入 | 1〜2日 |
| パイロット部署での運用テスト | 1〜2週間 |
